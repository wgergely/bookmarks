items#

Definitions of the core bookmark, asset, file item list widgets and their respective models.

Each list item widget derives from views.ThreadedBaseWidget, a heavily customized QListView widget. The views are rendered by delegate.BaseDelegate and data is served by models.BaseModel instances.

Hint

To customize which items are read and how they’re displayed, take a look at models.BaseModel.item_generator() and models.BaseModel.init_data() methods and their implementation in the item models.

models#

Defines the base model used to display bookmark, asset and files items.

The model loads and reset data via the BaseModel.init_data() and BaseModel.reset_data() methods.

The model itself does not store data, but all retrieved data is cached independently to bookmarks.common.item_data. The interface for getting and setting data can be found in the bookmarks.common.data. However, the model is responsible for populating the data cache. See BaseModel.item_generator().

The model exposes different datasets to the view using the BaseModel.task() and BaseModel.data_type() switches. This is because file items are stored as sequence and individual items simultaneously and because the file model also keeps separate data for each task folder it encounters.

The currently exposed data can be retrieved by BaseModel.model_data(). To change/set the data set emit the BaseModel.taskFolderChanged and BaseModel.dataTypeChanged signals with their appropriate arguments.

Worth keeping in mind that Bookmarks loads data in two passes. The model is responsible for discovering items, but will not populate all item the data, and instead uses helper threads to offload work. The model’s associated threads are defined by overriding BaseModel.queues.

Each base model is nested inside a QSortFilterProxy used for filtering _only_. Sorting however is implemented separately to run directly over the source data.

views#

The base view used to display bookmark, asset and file items.

The view uses BaseModel for getting the item data, BaseDelegate to paint the items.

The base list is a customised QListView, BaseInlineIconWidget adds inline icon capabilities, and ThreadedBaseWidget implements data updating with the helper threads.

delegate#

The delegate used to draw bookmark, asset and file items.

The list widgets have a number of custom features, such as clickable in-line icons, folder names, custom thumbnails that the delegate implements. All items are made up of a series of rectangles that the pain methods use to place drawn elements. See get_rectangles().

The downside of painting everything manually is that the paint performance is poor. For this reason the delegate tries to cache as much of the information as possible.

class bookmarks.items.file_items.FilesModel(parent=None)[source]#

Model used to list files in an asset.

The model will load files from one task folder at any given time. The current task folder can be retrieved by task(). Switching tasks is done via emitting the taskFolderChanged signals.

The model will load the found files into two separate data sets, one listing files individually, the other groups them into sequences. See bookmarks.common.sequences for the rules that determine how sequence items are identified.

Switching between FileItems and SequenceItems is done by emitting the dataTypeChanged signal.

Note

The model won’t necessarily load all files it encounters. If the parent bookmark item has a valid token config set, certain file extension might be excluded. See the bookmarks.tokens.tokens for details.

init_data(**kwargs)#

The main method used by the model to fetch and store item information from the file system.

The model itself does not store any data, instead, we’re using the datacache module to store item data.

The individual items are returned by item_generator().

reset_data(*args, force=False, emit_active=True)[source]#

Resets the model’s internal data.

The underlying data is cached in the datacache module, so here we’ll make sure the data is available for the model to use. When the optional force flag is set, we’ll use init_data to load the item data from disk.

Otherwise, the method will check if our cached data is available and if not, uses init_data to fetch it.
